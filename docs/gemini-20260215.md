# M2DX-Core 提案書レビュー (2026-02-15)

## 1. 概要
本ドキュメントは、Swift製FM音源ライブラリ「M2DX-Core」の提案書 (`docs/proposal.md`) に対する技術レビュー結果です。
DX7 (YM21280) の音源エンジンを現代的なSwift環境で再現し、かつMITライセンスで提供するという目標は、iOS/macOSの音楽アプリ開発において非常に価値が高いものです。

## 2. アーキテクチャの評価

### 技術的実現可能性: 高い
*   **Swift 6 Strict Concurrency**: オーディオ処理におけるスレッド安全性は最重要課題です。`swift-atomics` を用いた Lock-free SPSC リングバッファによるパラメータ渡しは、オーディオスレッドのブロックを回避する現代的かつ適切な設計です。
*   **数値演算**: Int32固定小数点演算によるビットアキュレート再現（DX7モード）と、Float32による拡張性（Cleanモード）の2つのパスを用意する設計は、「再現性」と「実用性」の両立において理にかなっています。
*   **SIMD / Accelerate**: `vDSP` を用いたバッチ処理（ダウンサンプリング、ミキシングなど）は、Swiftのパフォーマンスを最大限に引き出すために不可欠であり、適切な選定です。

### 仕様の完全性: 良好
*   **ディレクトリ構造**: `Sources` と `Tests` の構成が明確で、責務の分離（Engine, Tables, DSP, Infrastructure）が適切になされています。
*   **依存関係**: 外部依存を `swift-atomics` のみに絞り、システム標準の `Accelerate` を活用する方針は、ライブラリとしてのポータビリティとメンテナンス性を高めます。

## 3. リスクと課題

### パフォーマンス (Swiftのオーバーヘッド)
*   **ARCとバウンズチェック**: Swiftは安全性重視の言語であるため、配列アクセス時のバウンズチェックやARC（自動参照カウント）のオーバーヘッドがリアルタイムオーディオ処理においてボトルネックになる可能性があります。
    *   **対策**: クリティカルなDSPループ内では `UnsafeMutableBufferPointer` 等を活用し、不要なリファレンスカウント操作が発生しないよう、`@inlinable` や `final` クラスの活用、あるいはStructの適切な利用を徹底する必要があります。「Zero-Allocation」を掲げていますが、意図しない暗黙のアロケーションにも注意が必要です。

### クリーンルーム実装の徹底
*   **ライセンス汚染の回避**: msfa (C++) 等の既存実装からの「移行」フェーズにおいて、GPLやApache 2.0のコードが混入しないよう細心の注意が必要です。
    *   **評価**: 提案書にある「ルックアップテーブルを数式から自己生成する」というアプローチは、知的財産権のリスクを低減する上で非常に重要かつ正しい方針です。この方針をコード全体（特にエンベロープの挙動など）で徹底することが求められます。

### 再現性の検証
*   **聴感上の差異**: 数値的にビットアキュレートであっても、DAC部分やアナログ回路の挙動を含まない場合、実機と聴感が異なる場合があります。
    *   **推奨**: 単体テスト（数値検証）に加え、実際の波形比較や聴感テストのプロセスを確立することが重要です。ロードマップにある「Test suite」の充実はプロジェクトの成功に直結します。

## 4. ドキュメントの品質
*   非常に高品質です。既存のソリューション（msfa, Dexed）との比較が明確であり、採用する技術スタックの選定理由も論理的に説明されています。
*   ロードマップも具体的で、フェーズごとのゴールが明確です。

## 5. 結論
提案されたアーキテクチャと計画は技術的に妥当であり、Swiftエコシステムにおける強力なオーディオエンジンの基盤となると期待できます。
実装フェーズでは、特にSwift特有のパフォーマンス特性（ARC、バウンズチェック）への対策と、徹底したクリーンルーム実装の管理が成功の鍵となります。
